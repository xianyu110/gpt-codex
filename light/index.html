02 核心概念

理解 Threads，才算真正会用 Codex

重点不是安装命令，而是这套工作组织方式：文件夹是工作区，Thread 是具体任务线。

两层结构

层级 名称 作用
第一层 文件夹 / 工作区 承载项目目录、资料和代码
第二层 Thread 围绕单一目标推进的任务对话线

为什么这样设计

  • 不同任务拆进不同 Thread，减少上下文串味
  • 文件和过程分离，查找路径更清晰
  • 中断后容易续写，不容易整锅乱掉
  • 适合一个项目里并行推进多个目标

黄金法则：同一个文件夹只做一个大方向，同一个 Thread 只推进一件具体的事。

03 项目管理

先把文件夹分清，再谈后续效率

建议先做好目录分层，例如学习资料、真实项目、沙盒和通用工具分开。

推荐结构

dev/ ├── Learning/ ├── notes/ ├── Projects/ ├── sandbox/ └── tools/

管理原则

  • 不要什么对话都乱开新 Thread
  • 真实开发项目集中放在 Projects
  • 工作区负责文件，Thread 负责目标推进
  • 前期分类做好，后面才不会越用越乱

04 功能配置

图形化能力，是 Codex 很适合新手的优势

Codex 不只是命令执行，还把定时任务、Skills、全局规则和权限配置做成了图形化入口。

定时任务

可定时巡检服务器、自动修错并通过机器人发送摘要。

Skills

支持图形化管理技能，比纯命令行方式更容易理解与维护。

Skill Creator

可直接通过自然语言创建技能，对新手非常友好。

Personalization

相当于图形界面的 `AGENTS.md`，可设置全局规则与工作偏好。

MCP

给 Codex 接浏览器、数据库、GitHub、文件系统等外部工具和数据源。

记忆系统

沉淀长期偏好、输出格式和项目规则，让协作风格更稳定。

云端运行

适合本地不方便长时间运行、或希望异步推进的任务。

电脑自动化

可用于操作浏览器、检查页面、填表和处理重复动作，但要注意权限边界。

General 推荐

  • 保持电脑开机：打开
  • Follow-up behavior：`steer`

权限建议

  • 首页权限改成 `Full access`
  • 减少频繁确认，提升连贯执行效率

推理深度

  • 日常开发：`High`
  • 难活 / 硬活 / 大活：`Extra High`

05 使用方法

推荐工作流：先跑通小任务，再 Plan，再检查 diff

这一段按真实使用习惯组织，不再沿用旧页面的 CLI 教程写法。

STEP 0

第一次任务

先做一个小而可验证的任务：添加项目文件夹、新建 Thread、描述目标、确认计划、执行后预览结果。

打开 App -> 添加项目文件夹 -> 新建 Thread -> 描述小目标 -> 检查结果
STEP 1

Plan Mode

大型项目起始阶段先进入 `/plan`，只做规划，不急着写代码。等实现计划稳定后，再确认执行。

大型项目 -> 先规划 -> 形成规范与实现计划 -> 再正式开发
STEP 2

Status 与推理档位

用 `/status` 看当期用量；常规任务用 High，需要更稳时再拉到 Extra High。

/status Low / Medium / High / Extra High
STEP 3

语音 / 文字描述需求

复杂任务里可以直接“用嘴描述需求”，把要求讲清楚比堆 prompt 更重要。

STEP 4

前端要单独盯

前端效果要第一时间看；效果不好就重做，而不是将就。

STEP 5

代码管理

让 Codex 改代码前先说文件和计划；改完看 diff、跑最小检查，确认无误再提交。

先说改哪些文件 -> 看 git diff -> 跑最小检查 -> 再 commit
